## 3. Database schema (high level — full DDL in Phase 1)

### `config` schema
- `compensation_rates` — `(scheme_ref, key, value_numeric, value_json, effective_from, effective_to)` — JSON for rate scales (e.g., 50%→1,400 EGP table)
- `targets` — `(key, value, period, notes)` — Target-300 anchors
- `thresholds` — `(key, value, applies_to, notes)` — operational thresholds
- `salary_bands` — `(level, min_pct, max_pct, min_score_qualify)`
- `rating_bands` — `(min_score, max_score, label_en, label_ar, triggers_pip)`
- `cycle_periods` — `(id, type, label, start_date, end_date, status)`

### `def` schema
- `kpis` — 36 rows seeded, columns per Phase 0 confirmation
- `objectives` — 10 (6 company + 4 ops)
- `key_results` — 45 (28 company + 17 ops)
- `sops` — 10 SOPs
- `commission_schemes` — 8 schemes
- `roles` — primary roles + functional sub-roles + departments
- `kpi_role_weights` — role × kpi → weight
- `kpi_sop_links` — bidirectional KPI ↔ SOP
- `competencies` — appraisal competency catalog (per role)
- `users` — links auth.users to role/department/manager
- `departments` — 6+ (BD, AM, Ops, Onb, Marketing, Tech, Finance, HR, Mgmt)

### `track` schema
- `kpi_actuals` — period × employee × kpi → actual value, evidence
- `events` — transactional source events feeding KPI actuals (auction events, loading events, commission triggers, onboarding, referrals)
- `appraisal_cycles` — cycle id, type, period, status
- `appraisals` — cycle × employee → status + scores + sign-offs
- `appraisal_competency_scores` — per-appraisal × competency
- `commission_payouts` — calc snapshot per scheme/period/employee
- `payout_breakdown` — JSON detail of every input/formula/output line for the calc
- `pip_records` — triggered by red threshold

### `calc` schema (views + RPCs)
- `vw_kpi_score` — VIEW
- `vw_appraisal_quarterly_score` — VIEW
- `vw_appraisal_annual_score` — VIEW (auto-aggregates Q1–Q4 — closes the manual gap)
- `vw_okr_progress` — VIEW (cascade roll-up)
- `fn_run_commission(scheme_ref text, period_id uuid, employee_id uuid)` — RPC
- `fn_run_bonus_quarterly_opex(period_id uuid, employee_id uuid)` — RPC
- `fn_run_bonus_annual(year int, employee_id uuid)` — RPC

### `audit` schema
- `events` — actor, action, table_name, row_id, before_json, after_json, at

## 4. Edge Functions

### `crm-sync`
```
POST /functions/v1/crm-sync
{ payout_ids: [uuid] }
→ { ok: true, mode: 'dormant'|'live', synced: n, errors: [] }
```
Behavior:
- If `CRM_SUPABASE_URL` or `CRM_SUPABASE_ANON_KEY` unset → dormant return.
- Else: read each payout from this app's DB, push to CRM `commission_events` (idempotent on `payout_id`).

### `erp-sync`
Same dormant pattern. Two operations:
- `pull_employees` — refresh employee master from ERP
- `push_payroll` — export approved payouts to payroll batch

## 6. Formula engine

Three classes of formulas across the 36 KPIs:
1. **Ratio**: `actual / target` — most KPIs.
2. **Threshold**: `IF actual ≥ threshold THEN pass` — quality, on-time, fill rate KPIs.
3. **Composite**: combinations (e.g., GP Kicker = `accounts ≥ 13 AND gp ≥ 300_000`).

Formula stored as JSON `{ type, inputs[], expression, threshold? }` in `def.kpis.formula_engine_ref`. Pure JS evaluator in `utils/formula.js`. Mirrored as Postgres function for server-side calc.

## 7. Auto-aggregation Q1–Q4 → Annual

`calc.vw_appraisal_annual_score` joins:
- 4 quarterly appraisals for employee in year
- per-quarter weighted KPI score
- competency average
- manager rating

Output: `(kpi_avg * 0.7) + (competency_avg * 0.2) + (manager_avg * 0.1)` → rating band lookup → final rating.

If any quarter missing: status = `incomplete`, banner in Annual UI.

## 8. Bidirectional SOP↔KPI

`def.kpi_sop_links(kpi_id, sop_id)` indexed both ways. Two views:
- `calc.vw_kpis_by_sop` — list of KPIs per SOP
- `calc.vw_sops_by_kpi` — list of SOPs per KPI (usually 1:1 but link table allows N:M)

## 9. Audit pattern

Every table in `def` and `track` has trigger `_audit_trigger` calling `audit.log_change()` which writes:
```
(actor_user_id, action 'INSERT'|'UPDATE'|'DELETE', schema_name, table_name, row_pk, before_json, after_json, at = now())
```
HR/Admin/Finance/C-Level can SELECT from `audit.events`. No one can DELETE from `audit.events` (no policy granted).

## 10. Bilingual + RTL

- `lang/ar.js` + `lang/en.js` keyed by feature path (e.g., `okr.tree.title`).
- `LangContext` exposes `t(key)`, `lang`, `setLang`. Language stored in `localStorage('mrkoon_lang')`.
- On lang change: `<html dir="rtl|ltr" lang="ar|en">` updates immediately. Tailwind `rtl:` variants applied.
- All KPI/OKR/SOP/competency/rating-band content has `name_en` and `name_ar` columns at schema level.

## 11. Single Source of Truth pattern

- All values live in `config.*` tables.
- All calculators read via parameterized RPC, never inline.
- Admin UI edits `config.*` rows; immediate effect (no code deploy).
- Audit captures every config change.

## 12. Failure modes and fallbacks

- Supabase down → app shows offline banner, last-fetched data from cache (TanStack Query stale-while-revalidate).
- CRM/ERP dormant → "Integration not connected" badge in Finance dashboard, app fully usable.
- Auth expired → redirect to login with return URL.
- RLS policy reject → empty state with "you don't have access" copy, log to console for Admin debug.
