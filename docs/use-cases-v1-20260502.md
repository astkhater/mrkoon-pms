## UC-001: Set individual OKRs (period start)
**Actor**: Employee, Manager
**Precondition**: New OKR period open by Admin/HR; employee linked to a manager and department; department OKRs set.
**Trigger**: Employee navigates to "My OKRs" → "Set OKRs for [period]".
**Main flow**:
1. Employee enters Objective text (EN + AR).
2. Adds 2–5 Key Results (text, target value, unit, weight).
3. Links each KR to a parent Department KR.
4. Saves as draft.
5. Submits for manager approval.
6. Manager opens "Pending OKR approvals", reviews, approves or requests changes.
7. On approval, OKRs lock for the period.
**Exceptions**:
- Manager rejects → employee revises and resubmits.
- KR not linked to dept KR → save blocked with message.
**Postcondition**: Employee OKRs locked for period; visible in cascade tree.

## UC-002: Enter KPI actual (recurring)
**Actor**: Employee
**Precondition**: KPI assigned to employee for current period; period open.
**Trigger**: Frequency due (monthly/quarterly).
**Main flow**:
1. Employee opens "My KPIs" → sees KPIs needing entry (highlighted).
2. Selects a KPI; enters actual value, attaches evidence (link/file ref optional).
3. Saves.
4. System computes ratio score automatically; updates traffic light.
**Exceptions**:
- Actual outside reasonable range → warning + override comment required.
- Period closed → entry blocked.
**Postcondition**: `track.kpi_actuals` row inserted; audit logged; dashboard updated.

## UC-004: Quarterly review
**Actor**: Employee, Manager, Department Head
**Precondition**: Quarterly cycle open by HR; KPI actuals captured for the quarter.
**Trigger**: Quarter end.
**Main flow**:
1. Employee opens quarterly form, sees KPIs in scope (per role weights).
2. For each KPI, enters self-rating (anchored to actual already captured).
3. Rates each competency.
4. Submits.
5. Manager opens, rates each KPI and competency, comments.
6. Manager submits to Department Head.
7. Dept Head opens calibration view: all team ratings visible.
8. Dept Head approves or requests adjustments.
9. On approval, quarterly score locks, feeds annual aggregator.
**Exceptions**:
- Rating below "Meets" without comment → submit blocked.
- Calibration shows extreme inflation → Dept Head returns for re-rating.
**Postcondition**: Quarterly appraisal closed; row in `track.appraisals`; bonus calc unlocked for OpEx employees.

## UC-005: Annual appraisal
**Actor**: Employee, Manager, Department Head, HR
**Precondition**: All 4 quarterly appraisals completed for the year.
**Trigger**: Annual cycle open.
**Main flow**:
1. System auto-aggregates Q1–Q4 → KPI 70% + Competency 20% + Manager 10% → final rating.
2. Employee reviews aggregated form, adds annual self-reflection.
3. Manager adds annual comment.
4. Department Head reviews and approves.
5. HR final sign-off.
6. On sign-off: rating locks; annual bonus trigger fires; if rating < 2.5 → PIP record created.
**Exceptions**:
- Quarter missing → annual marked `incomplete`, banner shown until backfilled.
- Below 2.5 → PIP module triggered, separate workflow (out of scope of standard appraisal flow).
**Postcondition**: Annual appraisal closed; bonus run scheduled.

## UC-006: Run BD commission (quarterly)
**Actor**: Finance
**Precondition**: Quarter closed; BD reps' account counts and realized GP captured in `track.events`.
**Trigger**: Finance opens "Commission runs" → "Run BD-COMM-Q-v7 for [period]".
**Main flow**:
1. System fetches each BD rep's account count and realized GP for the quarter.
2. Looks up rate scale from `config.compensation_rates` for `BD-COMM-Q-v7`.
3. Calculates: per-account rate × accounts (capped at 30) → base commission.
4. Evaluates GP Kicker: accounts ≥ 13 AND GP ≥ 300,000 → kicker = 5% × realized GP, else 0.
5. Records breakdown line by line in `track.payout_breakdown`.
6. Saves run as `pending_approval`.
7. Finance reviews each rep's breakdown.
8. Finance approves run → status = `approved`.
9. Approval triggers `crm-sync` Edge Function (dormant → no-op until activated).
**Exceptions**:
- Missing GP data for a rep → run includes warning row, kicker not evaluated for that rep.
- Approval rejected → run reset to `draft` for re-run.
**Postcondition**: Approved payouts recorded; CRM sync attempted; export available for payroll.

## UC-007: Run AM commission (monthly, 6-component)
**Actor**: Finance
**Precondition**: Month closed; events captured (active clients, reactivations, upsells, referrals, txn counts).
**Trigger**: Finance triggers monthly AM run.
**Main flow**:
1. For each AM rep:
   - Retention: 200 × active clients
   - Reactivation: 500 × events, capped 1,500/month, conditional on ≥2 retained months
   - Upsell: 300 × new categories
   - Referral: 1,000 × signed referrals
   - Volume bonus: 1,500 if portfolio avg ≥2 txns/client/month (derived from `track.events`)
   - Portfolio bonus: 1,000 if ≥40 clients AND ≥80% transacting
2. Sum components → payout.
3. Breakdown line per component.
4. Save → approve → sync.
**Postcondition**: Same as UC-006.

## UC-008: Run VM commission (per-auction, monthly)
**Actor**: Finance
**Precondition**: Month closed; auction events with bidder counts and merchant attendance captured.
**Trigger**: Finance triggers monthly VM run.
**Main flow**:
1. For each auction in month:
   - If ≥1 bidder AND merchant attendance ≥ 6: 500 + 50×(bidders−1) → success payout
   - Else (no bidder OR attendance < 6): 250 → non-success flat
2. Sum across VM rep's auctions.
3. Save → approve → sync.
**Postcondition**: Same.

## UC-012: Quarterly OpEx bonus
**Actor**: Finance
**Precondition**: Quarterly appraisal closed and approved; salary band assigned.
**Trigger**: Finance triggers quarterly OpEx run.
**Main flow**:
1. For each OpEx employee with quarterly score ≥ 3.0:
   - Look up salary level (L1..L5)
   - Look up band % range (e.g., L3 → 10–20%)
   - Apply linear scaling within band based on score (e.g., score 3.5 → mid of band)
   - Multiply by salary → bonus
2. Score < 3.0 → no bonus.
**Postcondition**: Same.
