# Technical Requirements Document (TRD) — Mrkoon OKR/KPI/Appraisal Web App
Version: 1
Date: 2026-05-02

## 1. Stack
- **Frontend**: React 18 + Vite 5 + Tailwind CSS 3
- **Backend**: Supabase — PostgreSQL 15 + Auth + RLS + Edge Functions (Deno)
- **Hosting**: Netlify (production + staging branches)
- **Charts**: Chart.js 4 (matches Mrkoon CRM v15)
- **Language**: TypeScript-friendly JavaScript (CRM v15 uses .jsx; same here)
- **State**: React Context (Auth, Language) + TanStack Query for server data
- **Routing**: React Router v6
- **Form validation**: react-hook-form + zod
- **Date**: date-fns (Hijri optional via @hijri/calendar)
- **Testing** (project v2 onward per build-standards): Vitest + Playwright

## 2. Authentication
- Supabase Auth.
- Email magic link primary; password optional fallback.
- Session JWT carries `role`, `functional_role`, `department_id`, `manager_id` claims via custom JWT hook.
- Role detection on login → routes to role-specific dashboard.
- Protected routes via `<RoleGate roles={[...]}>`.
- SSO design hooks present (custom claims), activated when ERP/identity provider connects.
- Password reset flow: standard Supabase magic link.

## 3. Data architecture
Four Postgres schemas mirroring the four conceptual layers:
- `config` — values
- `def` — definitions
- `calc` — views and RPCs (computed)
- `track` — actuals and forms
- Plus `audit` — immutable event log

Every table has: `created_at`, `updated_at`, `created_by`, `updated_by`. Triggers populate `updated_*` and write to `audit.events` on INSERT/UPDATE/DELETE.

## 4. Row-Level Security
- Enabled on every table in `def`, `track`, `audit`, and on read-only views in `calc`.
- Policies per role, defined in `database/rls-policies/`.
- Helper functions: `auth.user_role()`, `auth.user_department()`, `auth.is_admin()`, `auth.manages(employee_id)`, `auth.is_finance()`, `auth.is_hr()`, `auth.is_clevel()`.
- Default deny — no SELECT or write succeeds without an explicit policy match.
- Phase 1 must pass isolation test (each role logs in, sees only permitted rows) before Phase 2 begins.

## 5. Integrations
### CRM (dormant)
- Edge Function `crm-sync`.
- Triggered on `track.commission_payouts` finalize.
- Reads `CRM_SUPABASE_URL` and `CRM_SUPABASE_ANON_KEY` from env.
- If both unset: returns `{ ok: true, mode: 'dormant', synced: 0 }`.
- If set: posts payout events to CRM `commission_events` table.
- Idempotent via `payout_id` unique key on CRM side.

### ERP (dormant)
- Edge Function `erp-sync`.
- Same dormant pattern.
- Future use: payroll export, employee master pull, salary master pull.
- Reads `ERP_API_URL` and `ERP_API_KEY` from env.

## 6. Non-functional requirements
- **Performance**: dashboard loads ≤2s on 10 Mbit; KPI traffic-light query ≤200ms; SOP↔KPI lookup ≤200ms; 50 employees × 3 years KPI history performance-tested in Phase 8.
- **Availability**: best-effort on Supabase free tier (staging) and Netlify; 99% target on prod.
- **Security**: RLS at DB; HTTPS only; no PII in URLs; salary visible only to Admin and Finance.
- **Audit**: every INSERT/UPDATE/DELETE in `def` and `track` writes `audit.events` (actor, action, before, after, timestamp).
- **i18n**: all text in `lang/ar.js` + `lang/en.js`; no hardcoded UI strings; RTL applied at `<html dir>` and Tailwind class strategy (`rtl:`).
- **Responsiveness**: desktop ≥1280px primary; tablet 768–1279px functional; mobile 360–767px functional for appraisal forms, KPI entry, and dashboards.
- **Accessibility**: WCAG 2.1 AA target — keyboard nav, focus visible, color-not-only contrast on traffic lights, alt on icons.
- **Privacy**: no investor/equity data; no individual salaries in shared outputs (already enforced at RLS); employee relations cases referenced by ID only (separate module, not in scope of this build).

## 7. Environment variables

### Frontend (Netlify build env)
| Var | Description | Required |
|---|---|---|
| `VITE_SUPABASE_URL` | Supabase project URL | Yes |
| `VITE_SUPABASE_ANON_KEY` | Supabase anon public key | Yes |
| `VITE_APP_NAME` | App display name | Yes |
| `VITE_APP_ENV` | `production` / `staging` | Yes |
| `VITE_DEFAULT_LANG` | `en` / `ar` | Yes (default `ar`) |

### Edge Function secrets (Supabase)
| Var | Description | Required |
|---|---|---|
| `CRM_SUPABASE_URL` | Mrkoon CRM v15 Supabase URL | No (dormant if unset) |
| `CRM_SUPABASE_ANON_KEY` | CRM anon key | No (dormant if unset) |
| `ERP_API_URL` | ERP base URL | No (dormant if unset) |
| `ERP_API_KEY` | ERP API key | No (dormant if unset) |
| `SERVICE_ROLE_KEY` | Server-side key for cross-schema writes | Yes |

## 8. Deployment
- **Staging**: separate Supabase project (free tier) + Netlify staging branch deploy.
- **Production**: production Supabase project + Netlify main branch.
- Migrations applied via `supabase db push` (or SQL Editor copy-paste in Phase 1 to walk through with Khater).
- Edge Functions deployed via `supabase functions deploy <name>`.
- Netlify access: internal-only (basic auth or Netlify Identity gating in Phase 9).

## 9. Browser support
- Chrome, Edge, Firefox, Safari latest 2 versions.
- Mobile Safari 16+, Chrome mobile latest.
- No IE.

## 10. Logging and observability
- Supabase logs (built-in) for SQL, Auth, Edge Functions.
- Frontend errors → console + optional Sentry (later).
- `audit.events` is the application audit source (queryable by HR/Admin).
