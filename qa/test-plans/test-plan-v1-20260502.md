# QA Test Plan v1 — Mrkoon OKR/KPI/Appraisal Web App
Date: 2026-05-02
Phase 8 deliverable. Drafted at Phase 0 so Phase 1–7 work targets these gates.

## 1. Entry criteria
- All Phase 1–7 acceptance criteria PASS in development.
- Schema, RLS, seed deployed in staging.
- Edge Functions deployed (dormant CRM/ERP).

## 3. Test categories

### 3.1 Role isolation (CRITICAL — blocks Phase 2 if any fail)
For each role, log in as test user and attempt:
- SELECT on all `def`, `track`, `audit`, `config` tables — verify rows visible match permission matrix in SDD §5.
- INSERT/UPDATE/DELETE outside permission — verify rejected with empty result (RLS) not error leak.
- Cross-tenant: log in as Employee A, attempt to read Employee B's KPIs/appraisals/payouts → 0 rows.
- Salary visibility: log in as HR, attempt to query salary fields → 0 rows or PERMISSION DENIED.
- Audit immutability: log in as Admin, attempt DELETE on `audit.events` → rejected.

### 3.2 Bilingual + RTL
- Toggle EN→AR and back: every screen verified (login, all dashboards, OKR tree, KPI entry, all 3 appraisal forms, bonus breakdown, SOP detail, admin config, audit log).
- Hardcoded string scan: lint rule (or grep) for any literal user-visible string in components → 0 results.
- RTL integrity: for each screen in AR, confirm:
  - `<html dir="rtl" lang="ar">` set
  - icons mirror correctly
  - text alignment reverses
  - no text overflow
  - dropdown chevrons mirror
  - charts/legends respect direction
- Persistence: switch lang, refresh, lang persists.

### 3.3 Mobile responsiveness
Tested at 375×667 (iPhone SE), 414×896 (iPhone XR), 768×1024 (iPad portrait):
- Login fully usable.
- Employee dashboard scrolls cleanly, all cards reachable.
- Monthly check-in form: full submission flow possible.
- KPI entry: actual value entry + save possible.
- Quarterly self-assessment: per-KPI rating + competency rating + reflection submit possible.
- Bonus breakdown: scrollable, no horizontal scroll on cards.

## 4. Bug log workflow
- Each defect logged in `qa/bug-log/` with: id, severity (critical/high/medium/low), title, steps, expected, actual, screenshot/log, reporter.
- Critical/high block phase exit; medium/low can ship with deferred ticket if Khater approves.
- After fix → re-test → mark resolved → regression sweep.

## 5. Exit criteria

- Role isolation matrix 100% pass.
- Performance targets met.
- QA report saved to `qa/test-results/qa-final-[YYYYMMDD].md`.
